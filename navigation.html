Navigation & Stacking ​

Veyra UI uses a shared navigation controller for all current windows.

  • Arrow keys navigate NUI-focused menus.
  • Controller frontend controls are forwarded from Lua into the active UI layer.
  • Context menus can keep player movement enabled with keepInput = true and captureMouse = false.
  • Disabled list items are skipped by controller navigation.
  • Lists wrap at the first and last selectable item.

Stacked Blocking UI ​

Blocking exports such as OpenConfirm, OpenInput, and OpenContext can be opened while another Veyra UI window is already active. Veyra UI keeps a Lua-side stack that mirrors the NUI window stack.

When the top blocking UI resolves or is cancelled:

  • Only the top window closes.
  • The previous window stays open.
  • Focus, mouse capture, keep-input, and movement lock settings restore from the previous window.
  • Pending results below the top window remain active.

This is useful for flows such as selecting a menu item, asking for confirmation, then returning to the original menu if it should stay open. For that style of flow, open the original menu with closeOnSelect = false.

lua
AddEventHandler('og_ui:itemSelected', function(item)
    if item.value ~= 'delete_vehicle' then return end

    CreateThread(function()
        local result = exports.og_ui:OpenConfirm({
            title = 'Delete Vehicle',
            description = 'Are you sure?',
            confirmText = 'Delete',
            cancelText = 'Cancel',
            lockMovement = true,
            captureMouse = true,
            keepInput = false
        })

        if result and result.confirmed then
            -- Delete vehicle here
        end
    end)
end)

Usage Notes ​

  • Keep menu item value fields stable. Treat them as action IDs.
  • Use OpenContext for movement-friendly interaction prompts.
  • Use OpenMenu for larger systems such as jobs, shops, crafting, emotes, and admin tools.
  • Use OpenMenuAwait when one script flow should wait for a selected menu item.
  • Use OpenConfirm and OpenInput inside CreateThread because they wait for a result.
  • Use stacked blocking UI intentionally when a flow needs to return to the previous window.
